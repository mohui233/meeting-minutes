# 规则知识库 (Rules)

本文件仅定义“判定标准与红线”，不负责流程调度。

## 1. 事实与证据要求
- **忠于原文**：所有“共识”、“方案”、“行动”必须能在原文中找到依据。
- **严禁脑补**：如果原文表述含糊或信息不足，必须降级写为 `待确认`，绝不允许强行推断或编造结论。
- **客观表述**：去除个人的情绪化表达，仅保留客观的问题、分歧、决策和承诺。

## 2. 行文与格式规范
- **通用友好**：语言风格需严谨、简练、务实。避免使用过度互联网化的黑话，多使用“统筹”、“复用”、“对接”、“牵头”等规范用语。
- **责任明确**：涉及后续行动时，必须明确具体的牵头单位或责任人；若会议中未明确，需写明 `待确定牵头方`。
- **精简去噪**：剔除会议中的口语、寒暄和偏离主题的闲聊，直奔核心结论。

## 3. 深度分析标准
- **跨部门对标**：涉及多部门职责重叠时，提炼“现状对比”与“解决方向”，并填入【重复模块逐项对标】表格中。
- **落地方案**：【最终需要落实的修改方案】必须按模块拆分，清晰界定参与各方的具体任务，以及落实在文档上的具体修改要求。
- **技术优化**：强制采用“痛点 -> 方案 -> 收益”三段式逻辑，并且方案必须侧重短期内可落地的动作。
- **人员洞察**：必须直白指出“谁在焦虑什么”、“谁对什么感兴趣”，禁止泛泛而谈的官话。

## 4. 执行红线
- **禁止过程冗余**：最终只输出结构化的纪要结果，严禁向用户输出“我正在思考”、“我的推理如下”等中间分析日志。
- **全局唯一性**：长文本分段提炼后，同一事项只能保留最准确的一条表述，必须进行去重和合并。
- **内容实质性**：拒绝空洞，禁止仅输出“达成一致”而无具体事项。所有结论必须包含具体决议内容。
- **字段冲突处理优先级**：若出现信息冲突，按以下优先级处理：会议时间（通知时间 > 实际开始时间）、会议地点（具体房间号 > 大楼描述）、参会人员（全量合并所有提到的人名并去重）。
- **严守命名规则**：禁止输出泛泛的文件名（如 `meeting_minutes.md`），必须严格遵循 `<原文件名>-会议纪要.md` 的命名规范。

# 规则知识库 (Rules)

本文件仅定义判定标准与红线，不负责流程调度。

## 1. 事实与证据要求

- **忠于原文**：所有结论、决策、行动项必须能在原文中找到依据。
- **严禁脑补**：原文表述含糊或信息不足，必须降级写为 `待确认`，不得强行推断或编造。
- **客观表述**：去除情绪化表达，仅保留客观的问题、分歧、决策与承诺。

## 2. 行文与格式规范

- **语言风格**：严谨、简练、务实。多用"统筹、复用、对接、牵头"等规范用语，避免互联网黑话。
- **责任明确**：后续行动必须明确牵头单位或责任人；会议中未明确的，写 `待确定牵头方`。
- **精简去噪**：剔除口语、寒暄、偏离主题的闲聊，直奔核心结论。
- **干净标题**：输出章节标题简洁（如 `一、会议背景`），严禁附带 `【必选项】` 等内部标注。
- **背景篇幅**：会议背景 ≤ 3 句话。
- **去重红线**：一个议题只说一次。需求、共识、决策、技术要点统一收敛到意见总结章节，不拆分表述。

## 3. 会议类型判定

- **跨部门/外部对接会**：参会方含 2 个及以上外部单位/机构，以需求对齐、方案研讨、跨部门对标为主。
- **内部工作会/任务分工会**：仅内部团队参与，以工作安排、任务分工、排期讨论为主。
- **缺省策略**：无法判定时默认按跨部门对接会处理。

## 4. 章节过滤规则

编排器加载模板章节清单，按会议类型过滤后动态重编号：
- 保留标记为 `通用` 和匹配当前类型的章节，其余丢弃（不输出）。
- 重复编号规则：跨部门 5 章（背景→意见总结→修改方案→待办→后续），内部 5 章（目标→意见总结→排期→待办→后续）。
- 必选章节正常输出；可选章节若不涉及，保留标题写 `**本次不适用**`。

## 5. 意见总结写作标准

意见总结是唯一的事实出口，合并了议题讨论、需求对齐、共识结论、关键决策、技术分析与可优化方向。按议题逐条展开，每条结构：

1. 背景与分歧（≤ 2 句）
2. 最终结论
3. 决策结果 —— 跨部门会议附带 `状态 + 原文依据（时间戳或快照）`；内部工作会附带 `负责人`
4. 技术要点或优化方向（有则写，无则跳过）

禁止将同一议题的需求、共识、决策拆分到多个章节重复表述。

## 6. 执行红线

- **禁止过程冗余**：最终只输出结构化纪要，严禁输出"我正在思考""推理如下"等中间日志。
- **全局唯一性**：长文分段提炼后，同一事项仅保留最准确的一条表述，强制去重合并。
- **内容实质性**：拒绝空洞结论。所有结论必须包含具体决议内容，禁止仅写"达成一致"。
- **字段冲突处理**：会议时间（通知时间优先于实际开始时间）、会议地点（具体房间号优先于大楼描述）、参会人员（全量合并去重）。
- **严守命名规则**：禁止泛化文件名（如 `meeting_minutes.md`），必须遵循 `<原文件名>-会议纪要.md`。

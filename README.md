# 会议纪要 Skill (meeting-minutes)

以前整理纪要很耗时间——录音反复听，重点容易漏，长会半天就搭进去了。后来直接把转写稿丢给 AI 出纪要，看着完整，但人就在现场，一看就知道哪儿不对。

于是把这一整套做法沉淀成了 Skill。核心抓三件事：定骨架、捋要点、卡证据。

---

## 一、先定骨架，别让 AI 自由发挥

统一模板管住输出结构，按会议类型自动过滤章节——跨部门对接和内部工作会各出一套，每套不超过 5 章。标题不能删，不涉及就写"本次不适用"。转写稿走固定流程：切段、提炼、合并，不让模型自己猜结构。会议背景压到 3 句话以内。

## 二、先捋要点，再出稿，同一件事不说两遍

不让模型一上来就写整篇。长会按议题切段，每段提炼核心信息后清掉原文，只留结构化要点。最后全局去重，按类型过滤章节、动态编号出稿。

关键是：需求、共识、决策、技术要点全收进"意见总结"，一次说完，不拆到好几章来回讲。

## 三、拿证据卡住，再用质检兜底

定了条死规矩——所有结论必须能在原话里找到依据，找不到就写"待确认"。每个关键决策带上状态和原文出处。

光写规则不够，模型不一定听。所以末尾接了一步自动质检：只检查不补写，分三个维度——决策能否回指原文、必选章节是否齐全、行动项是否具备责任人和截止日。跑完纪要再人工过一遍，结合质检结果删改，直接就能发。

## 文件结构

```
meeting-minutes/
├── SKILL.md                       # 技能入口：触发方式、路由策略、交付约定
├── README.md                      # 本文件
├── commands/
│   ├── meeting-minutes.md         # 主编排器：输入路由、分段提炼、章节过滤、出稿
│   └── minutes-verify.md          # 质量闸门：事实核查、结构检查、行动项完整性
└── rules/
    ├── minutes-rules.md           # 底线约束：证据要求、类型判定、行文规范、执行红线
    └── minutes-template.md        # 统一输出大纲，按类型过滤后≤5章
```

---

## 核心能力

- **双模路由**：自动判定跨部门对接会或内部工作会，匹配对应的元信息字段和章节组合
- **精简出稿**：两种类型均≤5章，一个议题只说一次，杜绝章节间重复
- **防幻觉**：每条结论须能回指原文，关键决策附带状态与原文快照
- **自动质检**：出稿后自动核验，三级结论判定可否直接发布
- **长文分段**：超 2 万字按议题切段提炼，防止上下文溢出
- **全局去重**：分段后统一实体名，化解信息冲突
- **交付规范**：输出 `<原文件名>-会议纪要.md`，质检结果附末尾

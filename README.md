# 会议纪要 Skill (meeting-minutes)

将会议录音或文本快速转化为符合政务/业务标准的结构化会议纪要，采用"分层编排 + 证据约束"设计，内置防幻觉质检闸门。

## 设计思路

以前手动整理纪要很耗时间，录音反复听，重点容易漏。后来直接把转写稿丢给 AI 出纪要，看着完整，但人就在现场就知道哪里不对。于是本 Skill 围绕四个核心原则设计：

### 1. 先把纪要骨架定住，别让 prompt 临场发挥

纪要模板直接固定 10 个一级标题：6 块必写，4 块可选。标题不能删，不涉及就写"本次不适用"。转写稿也按固定流程切分、提炼、合并，不让模型自己猜结构。

### 2. 先捋清会议要点，再统一出稿

不让模型一上来直接写终稿。长会先按议题切段，每段先提炼"背景、需求、共识、分歧、方案、决策、行动项"这些核心信息，提炼完就清掉原文，只保留结构化要点。最后再统一合并、去重清洗，按固定模板出稿。

### 3. 先拿证据卡住，再用质检把关，把瞎编拦在出稿前

定了条死规矩：所有"共识""方案""行动项"，都必须能在原话里找到依据；找不到就写"待确认"。每个关键决策都带上状态和原文依据。

光写规则还不够，模型不一定真听。所以又加了一步自动质检——质检只负责检查，不负责补写，重点看四件事：决策能不能回指原话，标题全不全，行动项有没有责任人和截止日。

### 4. 材料多了，再把历史纪要做成检索库（可选）

用 bge-m3 把决策和行动项转成向量存进 SQLite。写新纪要时不是全量灌进去，而是先检索，只拿最相关的 3-5 条给模型参考，避免 token 越花越多。

## 文件结构

```
meeting-minutes/
├── SKILL.md                          # 技能入口：元信息、触发方式、路由策略、交付约定
├── README.md                         # 本文件
├── commands/
│   ├── meeting-minutes.md            # 主编排器：路由调度、分段提炼、上下文清理、出稿
│   └── minutes-verify.md             # 质量闸门：事实核对、结构检查、完整性检查
└── rules/
    ├── minutes-rules.md              # 规则知识库：事实证据要求、行文规范、执行红线
    └── minutes-template.md           # 纪要输出大纲模板：10 个一级标题的结构约束
```

## 触发方式

| 方式 | 说明 |
| :--- | :--- |
| `/meeting-minutes` 或 `/会议复盘` | 指令触发，可附带文件路径 |
| 直接上传录音/文档/粘贴文本 | 隐式触发 |

## 路由策略

| 输入类型 | 处理方式 |
| :--- | :--- |
| 音频 (`.wav/.m4a/.mp3/.flac/.aac`) | 调用本地 Whisper 转写后分析 |
| `.docx` | 调用 `docx` 技能 |
| `.pdf` | 调用 `pdf` 技能 |
| `.txt/.md/.srt/.vtt` | 直接读取 |

## 核心能力

- **结构化纪要**：严格按 10 个一级标题出稿，涵盖背景、需求确认、共识、修改方案、关键决策、待办事项、深度洞察等
- **防幻觉机制**：所有结论必须能在原文中找到依据，找不到就写"待确认"；关键决策附带状态与原文快照
- **自动质检**：出稿后自动触发核验，检查事实依据、结构完整性、行动项责任人/截止日，判定可否直接发布
- **长文分段**：超长会议自动按议题切段提炼，避免上下文溢出
- **实体消歧**：分段提炼后全局合并去重，统一实体名称，化解信息冲突
- **最终交付**：输出 `<原文件名>-会议纪要.md`，质检结果以注释形式附在纪要末尾
